# Security Design

## Threat model

terok-shield provides egress network filtering for rootless Podman containers
running untrusted AI coding agents. The agents have full control inside their
container but must not be able to reach arbitrary internet hosts.

Goals:
- Default-deny outbound connectivity
- Allowlist-based access to specific destinations (domains or IPs)
- Block all RFC1918/link-local traffic (prevent lateral movement)
- Audit log all allowed and denied connections
- Fail closed: if the firewall cannot be applied, the container must not start
  with unrestricted network access

Non-goals:
- Inbound filtering (containers are not exposed)
- Rate limiting or bandwidth shaping
- Application-layer inspection

## Security boundary: `nft.py`

All nftables rulesets are generated by `nft.py`. This file is the auditable
security boundary:

- **Zero non-stdlib imports** — only `ipaddress`, `re`, `textwrap`. Enforced by
  an AST import isolation test and bandit SAST scan.
- **Input validation** — `safe_ip()` validates all IPs/CIDRs against
  `ipaddress.IPv4Address`/`IPv4Network` before string interpolation. `safe_name()`
  validates container names against `[a-zA-Z0-9_-]`.
- **Structural ordering** — RFC1918 reject rules are placed *before* the
  per-container allow set in every generated chain. This ordering is load-bearing:
  even if an attacker controls the allow set contents, RFC1918 destinations remain
  blocked.
- **Self-verification** — `verify_ruleset()` checks that an applied ruleset has
  the expected invariants (drop policy, reject type, RFC1918 blocks, log prefixes).

## Firewall modes

### Standard mode

Uses OCI hooks to apply per-container nftables rules inside the container's own
network namespace. Each container gets an isolated firewall.

Chain order (output): loopback → established → DNS → gate port → RFC1918 block →
allow set → deny-all.

### Hardened mode

Uses a dedicated bridge network (`ctr-egress`) and applies nftables rules in
podman's rootless-netns. All container traffic traverses the bridge and is
filtered at the forward chain.

Chain order (forward): established → DNS → gate → RFC1918 block →
ICMP → intra-bridge → global allow set → deny-all.

Hardened mode requires both the bridge network and `dnsmasq` to be present.

## Allowlisting

Allowlists are `.txt` files with one entry per line — either a DNS domain name
or a raw IP/CIDR. Lines starting with `#` are comments.

The bundled defaults use DNS domain names because:
- Domain names are stable over time; IPs can change without notice
- Domain names are easier for humans to review and audit
- Resolution to IPs happens at setup/runtime via `dig`

Users can add custom profiles with raw IPs/CIDRs for cases where DNS is not
appropriate (e.g., internal infrastructure with static IPs).

Resolved IPs are cached with `st_mtime`-based freshness (default 1 hour).

## Fail-closed enforcement

The OCI hook is the critical enforcement point. If any step in the hook fails —
ruleset application, IP loading, self-verification — the container must not
proceed with unrestricted network access.

## Audit logging

All firewall events (setup, teardown, allow, deny) are written as JSON-lines to
per-container log files. Audit logging is best-effort: failures are silently
ignored to avoid blocking container operations.

nftables-level logging uses rate-limited `log prefix` rules:
- `TEROK_SHIELD_ALLOWED`: traffic hitting the allow set
- `TEROK_SHIELD_DENIED`: traffic rejected by the deny-all rule
- `TEROK_SHIELD_RFC1918`: traffic blocked by RFC1918 rules

## Integration with terok

terok imports `terok-shield` as a library dependency. A thin adapter
(`shield_adapter.py` in the terok repo) maps terok's `Project` model and config
to `terok_shield.ShieldConfig`. The shield lifecycle hooks (`pre_start`,
`post_start`, `pre_stop`) are called from terok's container runners.

terok never calls the `terok-shield` CLI directly — only the Python API.
